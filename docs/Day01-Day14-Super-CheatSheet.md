# Day01 — Python Object Model

## 一句话总结

一切皆对象；变量不是对象，而是绑定到对象引用的名字。

## 今天必须记住的 Top10

**1. 一切皆对象**
一句话解释：整数、字符串、列表、函数、类、模块都是对象。
为什么存在：统一的对象模型让框架能用同一套协议操作一切。
什么时候使用：理解框架为何能把函数当路由、把类当依赖。
真实开发例子：FastAPI 把路由函数当作对象注册。
一句代码：`type(len)  # <class 'builtin_function_or_method'>`
容易踩坑：以为变量「装着」对象，其实只是名字。
记忆口诀：万物皆对象，变量只是牌。

**2. Name → Reference → Object**
一句话解释：变量是名字，绑定到对象引用，不是对象本身。
为什么存在：解释赋值、共享、变异的底层行为。
什么时候使用：分析共享状态 bug 时。
真实开发例子：`b = a` 让两个名字指向同一对象。
一句代码：`a = [1]; b = a; b.append(2); print(a)  # [1, 2]`
容易踩坑：以为赋值会复制对象。
记忆口诀：名字→引用→对象，赋值只搬牌不搬牛。

**3. 对象三要素：identity / type / value**
一句话解释：每个对象都有身份、类型、值。
为什么存在：区分「是不是同一个」和「值相不相等」。
什么时候使用：调试身份 vs 相等问题。
真实开发例子：`is None` 判断身份，`== "active"` 判断值。
一句代码：`id(x), type(x)`
容易踩坑：混淆 identity 与 value。
记忆口诀：身份、类型、值，三位一体。

**4. `==` vs `is`**
一句话解释：`==` 比较值，`is` 比较身份。
为什么存在：值相等和同一对象是两个概念。
什么时候使用：`is` 只用于 `None`/单例判断。
真实开发例子：`if value is None:` 而非 `== None`。
一句代码：`a == b  # 值；a is b  # 身份`
容易踩坑：用 `is` 比较字符串或数字（依赖驻留，不可靠）。
记忆口诀：is 认人，== 认值。

**5. 可变默认参数陷阱**
一句话解释：默认值在函数定义时只求值一次。
为什么存在：Python 在定义时构造默认对象。
什么时候使用：任何带默认容器参数的函数。
真实开发例子：多个请求共享同一个默认 list，状态泄漏。
一句代码：`def f(items=None): items = items or []`
容易踩坑：`def f(items=[])` 跨调用共享。
记忆口诀：默认可变，状态外泄。

**6. 函数是一等对象**
一句话解释：函数可赋值、传参、返回、存进容器。
为什么存在：支撑装饰器、依赖注入、回调、工具注册。
什么时候使用：把行为当数据传递时。
真实开发例子：AI backend 把工具存成函数字典。
一句代码：`handler = normalize; handler(" X ")`
容易踩坑：`hello` 与 `hello()` 混淆（对象 vs 调用）。
记忆口诀：函数即数据，可传可存。

**7. Callable 对象与 `__call__`**
一句话解释：任何能加括号调用的对象都是 callable。
为什么存在：让「带配置的行为」也能像函数一样调用。
什么时候使用：行为需要携带配置时。
真实开发例子：`Prefixer("user:")` 实例可直接调用。
一句代码：`class P: def __call__(self, x): ...`
容易踩坑：忘记类本身也是 callable（调用即建实例）。
记忆口诀：有 `__call__`，实例即函数。

**8. 变量存引用，不存对象**
一句话解释：赋值复制引用，不一定复制对象。
为什么存在：性能与共享语义。
什么时候使用：判断是否会意外共享变异。
真实开发例子：两个请求持有同一 list 互相污染。
一句代码：`print(a is b)  # True`
容易踩坑：期望赋值隔离状态。
记忆口诀：赋值搬引用，共享要小心。

**9. 对象身份 `id()`**
一句话解释：`id()` 返回对象生命周期内的身份。
为什么存在：学习期验证两名字是否同一对象。
什么时候使用：调试共享状态。
真实开发例子：确认 helper 是否原地改了调用者的数据。
一句代码：`id(a) == id(b)`
容易踩坑：生产代码依赖打印 id 而非清晰所有权。
记忆口诀：id 相同即同一。

**10. 对象身份 ≠ 数据库身份**
一句话解释：Python 的 `id` 与数据库主键无关。
为什么存在：内存对象和持久化实体是两层。
什么时候使用：ORM / 序列化边界。
真实开发例子：两个内存对象可代表同一数据库行。
一句代码：`user_a.id_pk == user_b.id_pk` 判等，非 `is`。
容易踩坑：用 `is` 判断两个 ORM 对象「是不是同一用户」。
记忆口诀：内存身份非业务身份。

## 今天最重要的代码

```python
# ==（值）vs is（身份）
if value is None:        # 身份：只对 None/单例用 is
    ...
if status == "active":   # 值：普通比较用 ==
    ...

# 可变默认参数：坏 vs 好
def add_item_bad(item: str, items: list[str] = []) -> list[str]:
    items.append(item)          # 跨调用共享同一 list
    return items

def add_item_good(item: str, items: list[str] | None = None) -> list[str]:
    if items is None:
        items = []              # 每次调用新建
    items.append(item)
    return items

# 函数是对象；callable 对象
handler = add_item_good         # 赋值给名字
class Prefixer:
    def __init__(self, prefix: str) -> None:
        self.prefix = prefix
    def __call__(self, value: str) -> str:
        return f"{self.prefix}{value}"
```

## 今天开发口诀

1. 一切皆对象，变量只是绑定到引用的名字。
2. `is` 只用于 `None` 和单例，值比较一律用 `==`。
3. 默认参数在定义时求值一次，别用可变默认。
4. 需要新对象就返回它，不要指望重绑定影响调用者。
5. 函数、类、模块都是对象，可以传递和存储。
6. 类是 callable，调用类就是建实例。
7. 需要「带配置的行为」时用 `__call__` 的 callable 对象。
8. 不要把请求状态存进全局对象。
9. 内存对象身份 ≠ 数据库业务身份。
10. 明确对象所有权，比打印 `id()` 更重要。

## 今天最容易忘记的知识

1. **可变默认参数**：为何忘——只在定义时求值一次很反直觉；正确 `items=None` 再置空；错误 `items=[]`；避免——默认容器一律用 `None` 哨兵。
2. **`is` 比较字符串**：为何忘——小整数/短字符串驻留时偶尔为 True；正确 `s == "x"`；错误 `s is "x"`；避免——只对 None 用 is。
3. **`hello` vs `hello()`**：为何忘——都能写；正确 传对象用 `hello`；错误 传调用结果 `hello()`；避免——问自己「传行为还是传结果」。
4. **赋值不复制**：为何忘——像复制；正确 需隔离用 copy；错误 `b=a` 当复制；避免——记 Name→Ref→Object。
5. **类也是 callable**：为何忘——只想到函数；正确 `User()` 建实例；错误 以为要 `new`；避免——记「调用类=建实例」。
6. **`==None` 反模式**：为何忘——能跑；正确 `is None`；错误 `== None`；避免——linter 开启。
7. **全局对象存请求态**：为何忘——方便；正确 请求作用域；错误 模块级 dict 存用户数据；避免——依赖注入。
8. **helper 隐式变异**：为何忘——看不见；正确 明确 mutate 或返回新对象；错误 偷偷 append；避免——函数边界即所有权边界。
9. **identity vs value**：为何忘——常同时成立；正确 分清 is/==；错误 混用；避免——记对象三要素。
10. **`id()` 生产依赖**：为何忘——调试好用；正确 设计清晰所有权；错误 靠 id 判业务相等；避免——用业务主键。

## 今天在 AI Backend 中的位置

```text
        ↓ 支撑
函数对象 → FastAPI 路由处理器 / Depends 依赖
```

名字不是对象，名字指向对象——这是理解 Python 一切行为的第一块基石。

# Day02 — Mutable vs Immutable

## 一句话总结

赋值改变名字指向，变异改变对象本身；共享可变对象是后端 bug 的根源。

## 今天必须记住的 Top10

**1. 可变 vs 不可变**
一句话解释：可变对象创建后能改，不可变不能。
为什么存在：决定对象是否可安全共享。
什么时候使用：判断要不要复制/隔离。
真实开发例子：list/dict/set 可变，int/str/bool/tuple 不可变。
一句代码：`x = "a"; x += "b"  # 新建字符串`
容易踩坑：以为改变量名=改对象。
记忆口诀：可变能改身，不可变换名。

**2. `b = a` 不复制**
一句话解释：复制的是引用，两名字同一对象。
为什么存在：性能与共享语义。
什么时候使用：分析共享变异。
真实开发例子：两请求共享 list 互相污染。
一句代码：`b = a; b.append(3); print(a)  # 变了`
容易踩坑：当成复制。
记忆口诀：等号搬引用，不搬对象。

**3. `id()` 验证身份**
一句话解释：`id(a)==id(b)` 说明同一对象。
为什么存在：学习期确认共享。
什么时候使用：调试共享 bug。
真实开发例子：确认 helper 是否改了调用者数据。
一句代码：`print(id(a) == id(b))`
容易踩坑：生产依赖 id 而非清晰所有权。
记忆口诀：id 同即同一。

**4. `append()` 原地变异**
一句话解释：`append`/`+=` 原地改，所有引用可见。
为什么存在：容器需要高效原地操作。
什么时候使用：想让改动对所有引用可见。
真实开发例子：向共享缓冲追加数据。
一句代码：`a += [3]  # id 不变`
容易踩坑：以为 `+=` 会新建。
记忆口诀：append/+= 原地改。

**5. `a = a + [3]` 新建重绑**
一句话解释：`+` 建新 list 并重绑名字。
为什么存在：需要不影响他人的新对象。
什么时候使用：想隔离改动。
真实开发例子：返回新列表不动原数据。
一句代码：`a = a + [3]  # id 变，b 不变`
容易踩坑：期望 `+` 改到别的引用。
记忆口诀：`a=a+[]` 换新牌。

**6. 浅拷贝**
一句话解释：`copy.copy()` 只拷第一层，嵌套仍共享。
为什么存在：外层隔离足够时省成本。
什么时候使用：只需隔离外层容器。
真实开发例子：拷贝配置但嵌套 dict 仍被共享。
一句代码：`copy.copy(x)`
容易踩坑：以为嵌套也隔离了。
记忆口诀：浅拷只护壳。

**7. 深拷贝**
一句话解释：`copy.deepcopy()` 递归拷贝整棵树。
为什么存在：需要完全隔离。
什么时候使用：job 需与原数据彻底隔离。
真实开发例子：任务独立复制输入。
一句代码：`copy.deepcopy(x)`
容易踩坑：盲目 deepcopy 掩盖所有权不清 + 昂贵。
记忆口诀：深拷全隔离，代价高。

**8. Hashable vs Unhashable**
一句话解释：不可变才可 hash，才能当 dict key。
为什么存在：key 的 hash 必须稳定。
什么时候使用：设计缓存/去重 key。
真实开发例子：`{("user", 1): "active"}` 合法。
一句代码：`{["user"]: 1}  # TypeError`
容易踩坑：list/dict/set 不能当 key。
记忆口诀：能哈希才能当钥匙。

**9. tuple 不总能当 key**
一句话解释：tuple 内全部可 hash 才可 hash。
为什么存在：嵌套可变则整体不稳定。
什么时候使用：设计复合 key。
真实开发例子：`("user", [1,2])` 非法。
一句代码：`("user", 1)` 合法。
容易踩坑：tuple 里塞 list。
记忆口诀：元组内含变则废。

**10. 谁拥有这个对象？**
一句话解释：真正的问题不是可变性，是所有权。
为什么存在：共享可变态跨请求/job/会话泄漏。
什么时候使用：设计服务层状态。
真实开发例子：用户会话 messages 不能全局共享。
一句代码：按 user/session 存储于 DB/cache。
容易踩坑：全局 list 存请求数据。
记忆口诀：先问所有权，再谈可变性。

## 今天最重要的代码

```python
import copy

# append/+= 原地改（所有引用可见）vs +  新建重绑（隔离）
a = [1, 2]; b = a
a += [3];        print(a, b)   # [1,2,3] [1,2,3]  同一对象
a = [1, 2]; b = a
a = a + [3];     print(a, b)   # [1,2,3] [1,2]    a 换新对象

# 浅拷贝 vs 深拷贝
original = [["python"]]
shallow = copy.copy(original);     shallow[0].append("x")   # 污染原对象
deep    = copy.deepcopy(original); deep[0].append("y")      # 不污染

# hashable key
cache = {("user", 1): "active"}      # OK：tuple 全可 hash
# cache = {("user", [1, 2]): "x"}    # TypeError：内含 list
```

## 今天开发口诀

1. 赋值改名字指向，变异改对象本身。
2. `b = a` 是共享引用，不是复制。
3. `append`/`+=` 原地改，`a = a + [...]` 新建重绑。
4. 只需外层隔离用浅拷贝，需要彻底隔离才用深拷贝。
5. 不要盲目 deepcopy 掩盖所有权不清。
6. 只有不可变对象可 hash、可当 dict key。
7. tuple 里含 list/dict 就不可 hash。
8. 缓存/去重 key 用稳定的 str/int/安全 tuple。
9. 请求/job/会话状态必须隔离，别全局共享。
10. 先问「谁拥有这个对象」，再决定复制或共享。

## 今天最容易忘记的知识

1. **`+=` 是原地**：为何忘——以为像 `+`；正确 知道 list `+=` 变异；错误 以为新建；避免——记 id 不变。
2. **浅拷贝嵌套共享**：为何忘——以为拷全了；正确 需要用 deepcopy；错误 只 copy() 就改嵌套；避免——画内存图。
3. **tuple 含 list 不可 hash**：为何忘——tuple 看着不可变；正确 全可 hash 才行；错误 复合 key 塞 list；避免——key 只用标量。
4. **deepcopy 昂贵**：为何忘——省事；正确 明确所有权；错误 到处 deepcopy；避免——按需隔离。
5. **str 不可变**：为何忘——`s += "x"` 像改；正确 知其新建；错误 期望原地；避免——记不可变清单。
6. **全局可变态泄漏**：为何忘——方便；正确 请求作用域/DB；错误 全局 dict；避免——依赖注入。
7. **`b=a` 共享**：为何忘——像复制；正确 需隔离用 copy；错误 当复制；避免——Name→Ref→Object。
8. **set/dict 不可当 key**：为何忘——只记 list；正确 用 frozenset；错误 用可变 key；避免——记可变=不可 hash。
9. **AI messages 共享**：为何忘——单人测试没事；正确 按 session 隔离；错误 全局 history；避免——scope by user。
10. **可变性 ≠ 所有权**：为何忘——只盯语法；正确 先设计所有权；错误 只问可不可变；避免——所有权优先。

上游承接 Day01 对象模型，下游直接决定并发安全与数据隔离。

## 一句话结束

赋值搬引用、变异改对象、共享要隔离——所有权是可变性问题的真正答案。

# Day03 — Functions & Parameter Passing

## 一句话总结

Python 按共享传递（call by sharing）：参数是指向同一对象的本地名字。

## 今天必须记住的 Top10

**1. 按对象引用传值 / call by sharing**
一句话解释：参数是本地名字，绑定到调用者传入的同一对象。
为什么存在：兼顾效率与共享语义，既非纯值传也非纯引用传。
什么时候使用：分析函数是否会改到调用者数据。
真实开发例子：helper 意外改了请求 payload。
一句代码：`def f(items): items.append(3)`
容易踩坑：以为是纯 pass-by-value。
记忆口诀：传的是引用副本，共享同一对象。

**2. 参数是本地名字**
一句话解释：调用时创建本地名字指向传入对象。
为什么存在：函数作用域隔离。
什么时候使用：理解重绑定为何不影响外部。
真实开发例子：函数内 `items = ...` 不动调用者。
一句代码：`items = items + [3]  # 只改本地名`
容易踩坑：以为重绑定会改调用者。
记忆口诀：参数是本地牌。

**3. 可变参数可被原地改**
一句话解释：mutable 参数在函数内 mutate，调用者可见。
为什么存在：共享同一对象。
什么时候使用：需要函数改调用者数据时（要显式）。
真实开发例子：`user["role"]="admin"` 改到外部 dict。
一句代码：`def add_role(u): u["role"]="admin"`
容易踩坑：无意的隐式变异。
记忆口诀：可变共享，改了都看见。

**4. 不可变参数不能原地改**
一句话解释：immutable 参数只能重绑本地名。
为什么存在：不可变对象无法原地改。
什么时候使用：传 int/str 时。
真实开发例子：`value = value + 1` 不影响调用者。
一句代码：`def inc(v): v = v + 1  # 无效于外部`
容易踩坑：期望函数改了外部 int。
记忆口诀：不可变，改不动外面。

**5. 变异 vs 重绑定**
一句话解释：变异改对象，重绑定改本地名指向。
为什么存在：区分两类「改」的可见性。
什么时候使用：判断调用者是否会看到变化。
真实开发例子：`append` 可见，`= items + []` 不可见。
一句代码：`items.append(3)` vs `items = items + [3]`
容易踩坑：混淆两者。
记忆口诀：变异共享见，重绑本地隐。

**6. `append()` vs `+`**
一句话解释：`append` 原地改，`+` 新建。
为什么存在：一个改共享对象，一个建新对象。
什么时候使用：决定改调用者还是返回新对象。
真实开发例子：想不动原数据就 `return items + [x]`。
一句代码：`return items + [3]`
容易踩坑：想隔离却用了 append。
记忆口诀：append 改原，+ 返新。

**7. 建了新对象就 return**
一句话解释：函数造新对象必须返回，调用者才能用。
为什么存在：重绑定不影响调用者。
什么时候使用：函数产出新状态时。
真实开发例子：构建新 messages 列表后返回。
一句代码：`def add(items): return items + [3]`
容易踩坑：造了新对象却不返回。
记忆口诀：新对象，记得还。

**8. 常见变异方法清单**
一句话解释：list 的 append/extend/sort 等会原地改。
为什么存在：容器提供原地操作。
什么时候使用：review 参数是否被改。
真实开发例子：`sort()` 改到共享列表顺序。
一句代码：`items.sort()  # 原地`
容易踩坑：以为 `sorted()` 与 `sort()` 一样。
记忆口诀：见到这些方法，警惕改原。

**9. 函数边界=所有权边界**
一句话解释：边界应说清函数是读、改还是返回新对象。
为什么存在：隐藏变异跨层制造 bug。
什么时候使用：设计服务层函数签名。
真实开发例子：service 函数明确 mutate or return。
一句代码：`-> None`（改）vs `-> list`（返新）。
容易踩坑：签名不表达意图。
记忆口诀：边界即所有权。

**10. 对 FastAPI/Playwright/AI 的影响**
一句话解释：可变参数会泄漏请求/页面/会话状态。
为什么存在：共享对象跨边界。
什么时候使用：所有传可变对象的地方。
真实开发例子：污染共享 messages 历史。
一句代码：隔离状态 + 返回新对象。
容易踩坑：全局/共享可变态。
记忆口诀：可变跨界，隔离先行。

## 今天最重要的代码

```python
# mutate（调用者可见）vs rebind（调用者不可见）
def mutate(items: list[int]) -> None:
    items.append(3)             # 改共享对象，外部可见

def rebind(items: list[int]) -> None:
    items = items + [3]         # 只改本地名，外部不变

# 造新对象就返回
def add_item(items: list[int]) -> list[int]:
    return items + [3]

# 不可变无法原地改
def increment(value: int) -> None:
    value = value + 1           # 对调用者无效
```

## 今天开发口诀

1. Python 是 call by sharing，传的是对象引用的副本。
2. 参数是本地名字，重绑定只改本地。
3. 可变对象在函数内 mutate，调用者会看见。
4. 不可变对象只能重绑本地名，改不到外部。
5. 变异改对象、重绑定改名字，可见性不同。
6. `append` 改原对象，`+` 建新对象。
7. 函数造了新对象，必须 return 出去。
8. 看到 append/extend/sort/pop 等，警惕原地改参数。
9. 函数边界要表达所有权：改它、读它、还是返新。
10. 传可变对象跨层前，先想清是否会泄漏状态。

## 今天最容易忘记的知识

1. **重绑不影响调用者**：为何忘——直觉像引用传；正确 造新对象 return；错误 以为 `items=...` 改外部；避免——记本地名。
2. **`+=` vs `+`**：为何忘——都是加；正确 隔离用 `+`；错误 隔离却用 `+=`；避免——记 `+=` 原地。
3. **int 改不动外部**：为何忘——像能改；正确 return 新值；错误 期望原地；避免——记不可变。
4. **隐式变异**：为何忘——看不见；正确 `-> None` 表意；错误 偷偷 append；避免——签名表达意图。
5. **sort vs sorted**：为何忘——名字像；正确 需保原用 sorted；错误 sort 改了共享；避免——记 sort 原地。
6. **造新对象不返回**：为何忘——以为改到外部；正确 return；错误 只在本地建；避免——检查返回值。
7. **dict.update 原地**：为何忘——像新建；正确 知其变异；错误 期望隔离；避免——记变异方法。
8. **共享 messages**：为何忘——单测没事；正确 传前复制/隔离；错误 直接改共享；避免——所有权明确。
9. **默认可变参数**：为何忘——Day01 已讲仍会犯；正确 None 哨兵；错误 `=[]`；避免——linter。
10. **call by sharing 名词**：为何忘——爱说 by reference/value；正确 说 call by sharing；错误 二选一；避免——记准确术语。

```text
        ↓
```

# Day04 — Scope & LEGB

## 一句话总结

名字查找按 Local→Enclosing→Global→Built-in；Python 用词法作用域。

## 今天必须记住的 Top10

**1. LEGB 查找顺序**
一句话解释：Local→Enclosing→Global→Built-in 依次找名字。
为什么存在：确定性的名字解析规则。
什么时候使用：分析名字从哪来。
真实开发例子：`len` 在 Built-in 层找到。
一句代码：局部→外层函数→模块→内建。
容易踩坑：以为全局优先。
记忆口诀：本地、外层、全局、内建。

**2. 词法作用域（lexical）**
一句话解释：按定义位置解析名字，不按调用位置。
为什么存在：可静态推理、易读。
什么时候使用：判断闭包捕获谁。
真实开发例子：函数捕获其定义处的外层变量。
一句代码：定义决定 enclosing，非 caller。
容易踩坑：以为调用者成了外层作用域。
记忆口诀：看定义，不看调用。

**3. `global`**
一句话解释：声明在模块全局作用域重绑名字。
为什么存在：函数内改模块级名字。
什么时候使用：极少数计数器/单例，谨慎。
真实开发例子：全局配置计数。
一句代码：`global count; count += 1`
容易踩坑：用它存请求状态。
记忆口诀：global 动模块，慎用。

**4. `nonlocal`**
一句话解释：声明在最近外层函数作用域重绑名字。
为什么存在：闭包内改外层函数变量。
什么时候使用：闭包状态需重绑时。
真实开发例子：计数器闭包 `count += 1`。
一句代码：`nonlocal count`
容易踩坑：以为 nonlocal 指全局。
记忆口诀：nonlocal 找外层，非全局。

**5. UnboundLocalError**
一句话解释：函数内赋值使名字变局部，读取时还没赋值。
为什么存在：局部性在编译期决定。
什么时候使用：调试此错误。
真实开发例子：`count = count + 1` 无声明报错。
一句代码：加 `global`/`nonlocal` 或换名。
容易踩坑：以为能直接读同名全局。
记忆口诀：函数内一赋值，名字即局部。

**6. 变异不需要声明**
一句话解释：`append` 变异对象不重绑名字，无需 global/nonlocal。
为什么存在：只有重绑名字才需声明。
什么时候使用：区分变异与重绑。
真实开发例子：`items.append(1)` 直接生效。
一句代码：`def add(): items.append(1)`
容易踩坑：以为变异也要 global。
记忆口诀：变异免声明，重绑才要。

**7. 遮蔽 shadowing**
一句话解释：内层同名会遮蔽外层。
为什么存在：局部优先。
什么时候使用：排查「变量怎么变了」。
真实开发例子：局部 `list` 遮蔽内建 `list`。
一句代码：别用 `list`/`id` 当变量名。
容易踩坑：遮蔽内建导致后续报错。
记忆口诀：同名内层遮外层。

**8. 闭包（预告）**
一句话解释：内层函数保留对定义处变量的访问。
为什么存在：外层返回后仍需状态。
什么时候使用：工厂函数/依赖工厂。
真实开发例子：`make_counter` 返回带状态的函数。
一句代码：见 Day05。
容易踩坑：捕获共享可变态。
记忆口诀：闭包捕获定义环境。

**9. 晚绑定 late binding**
一句话解释：闭包在调用时才查捕获变量的值。
为什么存在：捕获名字而非值。
什么时候使用：循环里造函数。
真实开发例子：循环建函数全返回最后值 `2 2 2`。
一句代码：`def f(i=i): return i` 修复。
容易踩坑：循环闭包共享循环变量。
记忆口诀：晚绑取值在调用时。

**10. 别把状态放全局**
一句话解释：全局状态会跨请求/job/会话泄漏。
为什么存在：模块全局被所有导入者共享。
什么时候使用：设计状态归属。
真实开发例子：FastAPI 请求用户不能存全局。
一句代码：请求作用域 + 依赖注入。
容易踩坑：全局 dict 存用户态。
记忆口诀：状态归生命周期，别归全局。

## 今天最重要的代码

```python
count = 0
def add() -> None:
    global count          # 不写会 UnboundLocalError
    count += 1

def make_counter():
    count = 0
    def counter() -> int:
        nonlocal count    # 重绑外层函数变量
        count += 1
        return count
    return counter

# 变异无需声明；重绑才需要
items: list[int] = []
def append_one() -> None:
    items.append(1)       # 变异，不需 global

# 晚绑定修复
funcs = [(lambda i=i: i) for i in range(3)]   # 0,1,2 而非 2,2,2
```

## 今天开发口诀

1. 名字查找顺序永远是 LEGB。
2. Python 用词法作用域：看定义位置，不看调用位置。
3. 函数内对名字赋值，它就是局部变量。
4. 读取前赋值前的局部名 → UnboundLocalError。
5. 重绑模块名用 `global`，重绑外层函数名用 `nonlocal`。
6. 变异对象不需要 global/nonlocal，重绑才需要。
7. `nonlocal` 指最近外层函数，不是全局。
8. 别用 `list`/`id`/`dict` 等内建名当变量，避免遮蔽。
9. 闭包晚绑定：循环里用 `i=i` 固定当前值。
10. 请求/job/会话状态归属其生命周期，不要放全局。

## 今天最容易忘记的知识

1. **UnboundLocalError**：为何忘——直觉能读全局；正确 声明或换名；错误 `x = x + 1` 无声明；避免——记编译期判局部。
2. **nonlocal 非全局**：为何忘——名字像；正确 找外层函数；错误 当 global 用；避免——记「最近外层」。
3. **变异免声明**：为何忘——以为都要；正确 append 直接用；错误 加多余 global；避免——分清变异/重绑。
4. **词法非动态**：为何忘——凭直觉；正确 看定义；错误 以为 caller 是外层；避免——记 lexical。
5. **遮蔽内建**：为何忘——`list` 好用；正确 换名；错误 覆盖内建；避免——避开内建名。
6. **晚绑定循环**：为何忘——看着对；正确 `i=i`；错误 直接捕获；避免——默认参数固定值。
7. **global 存请求态**：为何忘——方便；正确 请求作用域；错误 全局变量；避免——依赖注入。
8. **模块全局被共享**：为何忘——像局部；正确 无状态；错误 模块级可变态；避免——工厂函数。
9. **LEGB 顺序记反**：为何忘——四个层；正确 L→E→G→B；错误 全局优先；避免——口诀。
10. **闭包捕获名字**：为何忘——以为捕值；正确 知捕名字晚查；错误 期望捕快照；避免——记晚绑定。

## 今天在 AI Backend 中的位置

```text
作用域/LEGB（状态从哪来）
        ↓
FastAPI：Depends 依赖、请求生命周期、DB session 作用域
        ↓
Playwright：Page 不放全局，job 状态归 context
        ↓

        ↓
生产环境：状态归生命周期，杜绝隐藏全局态
```

名字按 LEGB 查、按定义处解析——控制好作用域，就控制好了状态的来源。

# Day05 — Closures

## 一句话总结

闭包 = 函数对象 + 捕获环境；用来保存状态而不依赖全局变量。

## 今天必须记住的 Top10

**1. 闭包定义**
一句话解释：闭包 = 函数对象 + 捕获环境。
为什么存在：外层返回后内层仍需访问其变量。
什么时候使用：需要携带配置/状态的函数。
真实开发例子：`make_counter()` 返回记数函数。
一句代码：`def outer(): x=0; def inner(): ...; return inner`
容易踩坑：以为「函数套函数」就是闭包。
记忆口诀：函数 + 捕获环境 = 闭包。

**2. 捕获环境**
一句话解释：闭包仍需要的外层名字集合。
为什么存在：延续定义处的变量。
什么时候使用：分析闭包依赖什么。
真实开发例子：`add_prefix` 捕获 `prefix`。
一句代码：`def mk(p): return lambda v: p+v`
容易踩坑：捕获了预期外的可变态。
记忆口诀：捕获的是名字环境。

**3. 状态保存**
一句话解释：闭包无需全局即可保存状态。
为什么存在：替代全局变量。
什么时候使用：轻量计数/累加。
真实开发例子：每个 counter 自带独立 count。
一句代码：`nonlocal count; count += 1`
容易踩坑：多个闭包共享同一可变态。
记忆口诀：各自闭包，各自状态。

**4. 工厂函数**
一句话解释：工厂函数生产「配置好的行为」。
为什么存在：分离配置与业务逻辑。
什么时候使用：需要参数化行为。
真实开发例子：`make_multiplier(3)` 生成乘 3 函数。
一句代码：`def mk(f): return lambda v: v*f`
容易踩坑：把配置写进全局。
记忆口诀：工厂造行为，配置和逻辑分家。

**5. 为何 `x = x + 1` 报 UnboundLocalError**
一句话解释：赋值使 x 变局部，读取时未赋值。
为什么存在：编译期决定局部性。
什么时候使用：调试闭包计数。
真实开发例子：漏写 nonlocal。
一句代码：`nonlocal x` 修复。
容易踩坑：忘 nonlocal。
记忆口诀：要重绑外层，先 nonlocal。

**6. 变异不需要 nonlocal**
一句话解释：变异捕获的可变对象无需 nonlocal。
为什么存在：没重绑名字。
什么时候使用：闭包里 append 到捕获的 list。
真实开发例子：累积日志到捕获列表。
一句代码：`logs.append(x)  # 无需 nonlocal`
容易踩坑：以为都要 nonlocal。
记忆口诀：变异免声明。

**7. 闭包 vs 类**
一句话解释：小配置用闭包，多状态多方法用类。
为什么存在：各有适用边界。
什么时候使用：状态需要名字/生命周期时用类。
真实开发例子：复杂领域对象用类。
一句代码：单行为小状态→闭包。
容易踩坑：用闭包硬塞复杂状态。

# Day06 — Decorators

## 一句话总结

装饰器是接收函数返回函数的函数；`@d` 等价 `func = d(func)`。

## 今天必须记住的 Top10

**1. 装饰器定义**
一句话解释：接收一个函数、返回一个函数的函数。
为什么存在：为横切关注点统一加行为。
什么时候使用：日志/计时/重试/鉴权/缓存。
真实开发例子：给多个接口统一加日志。
一句代码：`def d(func): def w(*a,**k): ...; return w`
容易踩坑：以为装饰器很魔法。
记忆口诀：吃函数、吐函数。

**2. `@d` 等价 `func = d(func)`**
一句话解释：语法糖，装饰后名字指向 wrapper。
为什么存在：简洁地重绑函数名。
什么时候使用：理解装饰器本质。
真实开发例子：`@trace` 就是 `f = trace(f)`。
一句代码：`@d\ndef f(): ...`
容易踩坑：不知道名字被替换成 wrapper。
记忆口诀：@ 就是重新赋值。

**3. wrapper 包装函数**
一句话解释：装饰后真正被调用的 callable。
为什么存在：在原函数前后插入行为。
什么时候使用：加前置/后置逻辑。
真实开发例子：before/after 打点。
一句代码：`def w(*a,**k): return func(*a,**k)`
容易踩坑：wrapper 忘记返回原结果。
记忆口诀：wrapper 才是真身。

**4. `*args, **kwargs` 转发**
一句话解释：用可变参数转发任意参数。
为什么存在：装饰器要通用。
什么时候使用：几乎所有生产装饰器。
真实开发例子：包装任意签名的接口。
一句代码：`def w(*args, **kwargs):`
容易踩坑：wrapper 不带 `*args/**kwargs` 导致 TypeError。
记忆口诀：通用装饰，星号转发。

**5. 返回原结果**
一句话解释：wrapper 必须返回 `func(*a,**k)`。
为什么存在：不能吞掉业务返回值。
什么时候使用：所有装饰器。
真实开发例子：接口返回值必须透传。
一句代码：`result = func(*a,**k); return result`
容易踩坑：忘 return，返回 None。
记忆口诀：算完要还回去。

**6. `functools.wraps`**
一句话解释：保留原函数元数据（`__name__`/`__doc__`/签名）。
为什么存在：日志、调试、框架依赖元数据。
什么时候使用：每个生产装饰器。
真实开发例子：不加则日志显示 `wrapper` 而非 `create_user`。
一句代码：`@wraps(func)`
容易踩坑：漏 wraps 破坏 FastAPI/inspect。
记忆口诀：装饰必带 wraps。

**7. 通用装饰器模板**
一句话解释：`@wraps` + `*args/**kwargs` + 返回结果。
为什么存在：覆盖多数场景。
什么时候使用：写新装饰器时照抄。
真实开发例子：见代码区模板。
一句代码：见「今天最重要的代码」。
容易踩坑：偏离模板漏项。
记忆口诀：模板照抄不出错。

**8. 横切关注点**
一句话解释：日志/计时/重试/鉴权/缓存/限流/tracing。
为什么存在：与业务正交、可复用。
什么时候使用：多处需要同一附加行为。
真实开发例子：AI token 统计装饰器。
一句代码：`@track_tokens`
容易踩坑：把业务逻辑塞进装饰器。
记忆口诀：横切正交，业务解耦。

**9. FastAPI 依赖装饰器**
一句话解释：FastAPI 用装饰器注册路由、依赖元数据。
为什么存在：声明式绑定 + 依赖注入。
什么时候使用：定义接口。
真实开发例子：`@router.post("/x")`。
一句代码：`@app.get("/")`
容易踩坑：自定义装饰器漏 wraps 破坏签名解析。
记忆口诀：路由靠装饰器声明。

**10. 装饰器常见 bug**
一句话解释：漏 `*args`、漏 return、漏 wraps、吞异常、重试非幂等、同步包异步。
为什么存在：模板项缺失。
什么时候使用：review 装饰器。
真实开发例子：重试了非幂等的写操作。
一句代码：对照 bug 清单自查。
容易踩坑：见上。
记忆口诀：无 wraps 的装饰器默认可疑。

## 今天最重要的代码

```python
from collections.abc import Callable
from functools import wraps
from typing import Any

def my_decorator(func: Callable[..., Any]) -> Callable[..., Any]:
    @wraps(func)                        # 保留元数据
    def wrapper(*args: Any, **kwargs: Any) -> Any:
        # before
        result = func(*args, **kwargs)  # 转发 + 保留返回值
        # after
        return result
    return wrapper

# 重试装饰器（Playwright / LLM 常用）
def retry(times: int = 3):
    def deco(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(times):
                try:
                    return func(*args, **kwargs)
                except Exception:
                    if attempt == times - 1:
                        raise
        return wrapper
    return deco
```

## 今天开发口诀

1. 装饰器就是「接收函数、返回函数」的函数。
2. `@d` 等价 `func = d(func)`，名字被替换成 wrapper。
3. wrapper 才是装饰后真正运行的 callable。
4. 用 `*args, **kwargs` 转发任意参数。
5. wrapper 必须 `return func(*args, **kwargs)`。
6. 每个生产装饰器都要 `@functools.wraps(func)`。
7. 无 wraps 的装饰器默认可疑，会破坏日志与框架。
8. 装饰器用于横切关注点，别塞业务逻辑。
9. 不要重试非幂等操作，不要静默吞异常。
10. 同步 wrapper 不能包装异步函数。

## 今天最容易忘记的知识

1. **漏 wraps**：为何忘——不加也能跑；正确 `@wraps(func)`；错误 裸 wrapper；避免——模板固定带上。
2. **漏 return**：为何忘——只关注副作用；正确 返回结果；错误 返回 None；避免——检查返回值。
3. **漏 `*args/**kwargs`**：为何忘——测试签名简单；正确 通用转发；错误 固定参数；避免——用模板。
4. **名字被替换**：为何忘——看不见；正确 知 `f` 已是 wrapper；错误 以为还是原函数；避免——理解 `f=d(f)`。
5. **吞异常**：为何忘——想「优雅」；正确 透传或转译；错误 静默 except；避免——只处理已知异常。
6. **重试非幂等**：为何忘——重试看着安全；正确 只重试幂等；错误 重试写操作；避免——判断幂等性。
7. **同步包异步**：为何忘——都是函数；正确 用 async wrapper；错误 sync 包 async；避免——区分协程。
8. **记录敏感参数**：为何忘——日志方便；正确 脱敏；错误 打印 prompt/密钥；避免——过滤敏感。
9. **带参装饰器三层**：为何忘——层数多；正确 `deco(args)→deco(func)→wrapper`；错误 少一层；避免——记三层结构。
10. **元数据用途**：为何忘——只想到名字；正确 签名/注解也依赖；错误 以为无所谓；避免——记 FastAPI 依赖它。

## 今天在 AI Backend 中的位置

```text
装饰器（横切关注点，闭包的应用）
        ↓
FastAPI：@router 注册路由、鉴权/日志装饰器
        ↓
Playwright：@retry 重试不稳定的自动化步骤
        ↓
AI Backend：@track_tokens / @trace 追踪 LLM 调用与工具
        ↓
生产环境：日志/计时/重试/限流统一、与业务解耦
```
上承 Day05 闭包，下启后续所有框架的声明式能力。

## 一句话结束

装饰器把横切关注点从业务里抽出来——记住 `@d` 就是 `func = d(func)`，且永远带上 `functools.wraps`。

# Day07 — Iterators & Generators

## 一句话总结

数据可共享、状态不可共享；生成器是可暂停可恢复、只能消费一次的迭代器。

## 今天必须记住的 Top10

**1. 可迭代 vs 迭代器**
一句话解释：iterable 能产出迭代器，iterator 逐个产值并记住位置。
为什么存在：容器可复用，遍历状态归迭代器。
什么时候使用：理解 for 循环底层。
真实开发例子：一个 list 可开多个独立迭代器。
一句代码：`it = iter([1,2]); next(it)`
容易踩坑：把「数据」和「遍历状态」混为一谈。
记忆口诀：容器产迭代器，迭代器记位置。

**2. `iter()` / `next()`**
一句话解释：`iter` 要迭代器，`next` 要下一个值。
为什么存在：迭代协议。
什么时候使用：手动遍历。
真实开发例子：逐条读流。
一句代码：`next(iter(["a","b"]))  # a`
容易踩坑：对已耗尽迭代器再 next。
记忆口诀：iter 拿器，next 拿值。

**3. StopIteration**
一句话解释：没有更多值时抛的结束信号。
为什么存在：区分「结束」与真实数据 `None`。
什么时候使用：迭代结束判断。
真实开发例子：for 内部捕获它结束循环。
一句代码：耗尽后 `next()` 抛 StopIteration。
容易踩坑：当成崩溃处理。
记忆口诀：StopIteration 是句号，不是错误。

**4. 生成器**
一句话解释：用 `yield` 的函数创建的迭代器。
为什么存在：惰性、可暂停的数据流。
什么时候使用：流式/大数据/管道。
真实开发例子：LLM token 流。
一句代码：`def g(): yield 1; yield 2`
容易踩坑：以为调用即执行函数体。
记忆口诀：yield 造生成器，惰性产值。

**5. `yield` vs `return`**
一句话解释：`return` 结束返回一个值，`yield` 产一个值并暂停。
为什么存在：可恢复的数据流。
什么时候使用：需要逐步产出。
真实开发例子：逐页抓取。
一句代码：`yield item`
容易踩坑：以为 yield 后函数就结束。
记忆口诀：return 落幕，yield 暂停。

**6. 生成器生命周期**
一句话解释：Created→Running→Paused→Running→Exhausted。
为什么存在：暂停/恢复模型。
什么时候使用：理解流状态。
真实开发例子：next 驱动到下一个 yield。
一句代码：`Created→yield 暂停→resume→耗尽`
容易踩坑：期望耗尽后重启。
记忆口诀：产完即止，不重来。

**7. 只能消费一次**
一句话解释：生成器耗尽后不重启。
为什么存在：状态是一次性的。
什么时候使用：需要复用就转 list。
真实开发例子：`list(gen)` 后再遍历为空。
一句代码：`list(gen); list(gen)  # [] `
容易踩坑：`list()`/`sum()` 调试时消费掉了。
记忆口诀：一次性水流，用完就干。

**8. 惰性求值**
一句话解释：按需产值，改善首字节时间与内存。
为什么存在：不必先算全部。
什么时候使用：大/延迟数据、流式。
真实开发例子：StreamingResponse 边算边发。
一句代码：`(x*2 for x in big)`
容易踩坑：以为只是省内存。
记忆口诀：惰性=按需，省内存又快首响。

**9. 生成器表达式 vs 列表推导**
一句话解释：`()` 惰性流，`[]` 立即全量。
为什么存在：流式 vs 索引/复用。
什么时候使用：需索引/长度/多次遍历用 list。
真实开发例子：一次性管道用 genexpr。
一句代码：`(...)` vs `[...]`
容易踩坑：需要复用却用了生成器。
记忆口诀：要复用用方括号，要流式用圆括号。

**10. `yield from`**
一句话解释：把产值委托给另一个可迭代。
为什么存在：组合生成器管道。
什么时候使用：拼接子生成器。
真实开发例子：串联多段流。
一句代码：`yield from child()`
容易踩坑：误对字符串逐字符迭代。
记忆口诀：yield from 转交产值。

## 今天最重要的代码

```python
# 迭代协议
it = iter([1, 2, 3]); print(next(it))     # 1

# 生成器：惰性、可暂停
def numbers():
    yield 1
    yield 2

gen = (x * 2 for x in range(3))            # 生成器表达式
print(list(gen))   # [0, 2, 4]
print(list(gen))   # []  已耗尽，只能消费一次

# yield from 组合管道
def child():
    yield "a"; yield "b"
def parent():
    yield from child()
    yield "c"
```

## 今天开发口诀

1. 数据可共享，状态不可共享。
2. iterable 产出 iterator，遍历状态归 iterator。
3. `StopIteration` 是结束信号，不是崩溃。
4. 为何不用 None 结束？因为 None 可能是真实数据。
5. 生成器是可暂停可恢复的一次性迭代器。
6. 生成器只能消费一次，需要复用就转 list。
7. 用 `list()`/`sum()` 调试会消费掉生成器。
8. 惰性求值改善的是内存 + 首字节时间。
9. 需索引/长度/复用用列表，需流式用生成器。
10. 流式失败可能发生在部分输出之后，清理要显式。

## 今天最容易忘记的知识

1. **只能消费一次**：为何忘——像 list；正确 需复用转 list；错误 二次遍历；避免——记一次性。
2. **调试消费生成器**：为何忘——随手 `list()`；正确 用 itertools.tee 或转 list；错误 打印后流空；避免——警惕消费。
3. **StopIteration 非错误**：为何忘——像异常；正确 当结束信号；错误 当崩溃；避免——理解协议。
4. **调用不执行体**：为何忘——像普通函数；正确 next 才跑；错误 期望立即执行；避免——记惰性。
5. **`sum()` 二次为 0**：为何忘——聚合看着无害；正确 先物化；错误 聚合两次；避免——记一次性。
6. **genexpr 复用**：为何忘——想省内存；正确 需复用用 list；错误 复用生成器；避免——按需选。
7. **yield from 字符串**：为何忘——字符串可迭代；正确 明确意图；错误 逐字符迭代；避免——检查类型。
8. **流式部分失败**：为何忘——以为原子；正确 处理半输出；错误 忽略中途异常；避免——显式错误处理。
9. **资源清理**：为何忘——生成器隐式；正确 显式 close/finally；错误 泄漏；避免——上下文管理（Day12）。
10. **共享生成器**：为何忘——像数据；正确 每消费者独立；错误 多方共享一个；避免——状态不共享。

## 今天在 AI Backend 中的位置

```text
迭代器/生成器（惰性一次性数据流）
        ↓
FastAPI：StreamingResponse 分块下发
        ↓
Playwright：逐页抓取管道
        ↓
AI Backend：LLM token 流，每个流独占自己的状态
        ↓
生产环境：数据共享、状态隔离，流式失败要显式处理
```
上承函数/闭包，下启 Day12 上下文管理器（流的资源清理）。

## 一句话结束

生成器是可暂停、只能喝一次的水流——数据可共享，遍历状态永远独占。

# Day08 — Exception Handling

## 一句话总结

异常处理是生产控制流：在能加意义的层捕获，让未知失败继续上抛。

## 今天必须记住的 Top10

**1. try/except 基本**
一句话解释：为已知失败路径兜底。
为什么存在：区分正常流与错误流。
什么时候使用：可预期的失败。
真实开发例子：解析失败回退默认。
一句代码：`try: risky() except SpecificError: recover()`
容易踩坑：用它掩盖未知 bug。
记忆口诀：try 兜已知，勿藏未知。

**2. 捕获具体异常**
一句话解释：只捕你知道怎么处理的异常。
为什么存在：保留错误含义。
什么时候使用：明确的失败类型。
真实开发例子：`except ZeroDivisionError`。
一句代码：`except ValueError:`
容易踩坑：业务里裸 `except Exception`。
记忆口诀：能处理才捕，其余放行。

**3. 异常控制流**
一句话解释：try 内出错后，该块后续行不执行。
为什么存在：立即跳转到 except。
什么时候使用：理解执行顺序。
真实开发例子：出错行后的日志不会打。
一句代码：`try: A; 1/0; B except: C`
容易踩坑：以为出错后还会往下走。
记忆口诀：出错即跳，后续不跑。

**4. 异常传播**
一句话解释：不捕获就沿调用栈上抛。
为什么存在：让更高层用更好上下文处理。
什么时候使用：本层无法妥善处理时。
真实开发例子：底层抛，API 层转 HTTP。
一句代码：不写 except 即传播。
容易踩坑：过早在底层吞掉。
记忆口诀：处理不了就上抛。

**5. `raise`**
一句话解释：无法安全继续时主动抛。
为什么存在：破坏不变量要中止。
什么时候使用：非法输入/失败依赖。
真实开发例子：`if age<0: raise ValueError`。
一句代码：`raise ValueError("...")`
容易踩坑：该抛却返回 None。
记忆口诀：走不下去就 raise。

**6. 返回 None vs raise**
一句话解释：预期缺失返 None，非法状态 raise。
为什么存在：区分「没有」和「错了」。
什么时候使用：查无结果 vs 输入非法。
真实开发例子：找不到用户返 None，非法参数 raise。
一句代码：`-> User | None` vs `raise`
容易踩坑：用 None 掩盖错误。
记忆口诀：无则 None，错则 raise。

**7. 自定义异常**
一句话解释：编码领域含义的异常类。
为什么存在：API 映射状态码、worker 决定重试。
什么时候使用：大型后端分类失败。
真实开发例子：`InvalidPromptError`/`LLMRequestError`。
一句代码：`class LLMRequestError(RuntimeError): pass`
容易踩坑：所有失败塌缩成 None。
记忆口诀：领域异常，失败可分类。

**8. 异常链 `raise ... from ...`**
一句话解释：保留根因同时转译为领域异常。
为什么存在：不丢失原始原因。
什么时候使用：底层异常转业务异常。
真实开发例子：`raise LLMRequestError(...) from timeout`。
一句代码：`raise DomainError(...) from err`
容易踩坑：转译时丢了根因。
记忆口诀：转译带 from，根因不丢。

**9. Traceback / 根因**
一句话解释：回溯回答哪个函数/哪行/什么异常/根因。
为什么存在：调试故事。
什么时候使用：排障。
真实开发例子：日志保留根因但脱敏。
一句代码：日志记异常类型 + 根因。
容易踩坑：日志泄漏敏感数据。
记忆口诀：留根因，去敏感。

**10. 框架错误模式**
一句话解释：FastAPI 边界抛 HTTPException，service 抛领域异常，Playwright 截图后清理再抛。
为什么存在：一致的失败转译。
什么时候使用：设计各层错误。
真实开发例子：区分可恢复超时与不可恢复登录失败。
一句代码：`raise HTTPException(status_code=400, ...)`
容易踩坑：内部 traceback 直接暴露给用户。
记忆口诀：边界转 HTTP，内部留领域。

## 今天最重要的代码

```python
# 具体异常 + 领域异常 + 异常链
class LLMRequestError(RuntimeError):
    pass

def call_llm(prompt: str):
    try:
        return provider.call(prompt)
    except TimeoutError as error:
        raise LLMRequestError("LLM request timed out") from error  # 保留根因

# 预期缺失返 None，非法状态 raise
def find_user(email: str) -> "User | None":
    ...
def check_age(age: int) -> None:
    if age < 0:
        raise ValueError("age must not be negative")

# FastAPI 边界转译
from fastapi import HTTPException
raise HTTPException(status_code=404, detail="not found")
```

## 今天开发口诀

1. 异常处理是生产控制流，不只是防崩溃。
2. 只捕获你知道怎么处理的具体异常。
3. 业务逻辑里避免裸 `except Exception`。
4. 处理不了就让异常向上传播。
5. 无法安全继续时用 `raise`。
6. 预期缺失返回 None，非法状态抛异常。
7. 大型系统用自定义异常编码领域含义。
8. 转译异常用 `raise ... from ...` 保留根因。
9. 日志保留根因但绝不泄漏敏感数据。
10. API 边界转 HTTPException，内部保持领域异常。

## 今天最容易忘记的知识

1. **过早捕获**：为何忘——想稳；正确 在能加意义处捕；错误 底层吞光；避免——就近有意义原则。
2. **裸 except**：为何忘——省事；正确 具体异常；错误 `except Exception`；避免——只捕已知。
3. **丢根因**：为何忘——转译图快；正确 `from err`；错误 直接 raise 新异常；避免——异常链。
4. **None 掩盖错误**：为何忘——不想崩；正确 非法则 raise；错误 一律返 None；避免——分清缺失/错误。
5. **泄漏 traceback**：为何忘——调试留着；正确 边界转 HTTP；错误 原始错误给用户；避免——转译边界。
6. **吞 Playwright 失败**：为何忘——想继续；正确 截图清理再抛；错误 静默 pass；避免——保留证据。
7. **区分可恢复**：为何忘——都当错；正确 超时可重试 vs 登录失败不可；错误 一刀切；避免——分类处理。
8. **控制流跳转**：为何忘——以为顺序执行；正确 知出错即跳 except；错误 期望继续；避免——记控制流。
9. **finally 清理**：为何忘——只想 except；正确 finally 释放资源；错误 泄漏；避免——Day12 上下文。
10. **自定义异常层级**：为何忘——用内建凑；正确 继承合适基类；错误 全 Exception；避免——领域建模。

## 今天在 AI Backend 中的位置

```text
异常处理（检测/路由/转译/解释失败）
        ↓
FastAPI：边界 HTTPException，避免暴露内部 traceback
        ↓
Playwright：截图→清理→重抛，区分可恢复超时
        ↓
AI Backend：InvalidPromptError / LLMRequestError / ToolExecutionError / RateLimitError
        ↓
生产环境：失败可分类、可搜索、根因可追、敏感数据不泄漏
```
上承生成器（流式失败），下启架构层的错误契约设计。

## 一句话结束

在能给失败增加意义的那一层捕获——具体捕获、领域转译、保留根因，让未知继续上抛。

# Day09 — Modules & Packages

## 一句话总结

导入即执行并缓存；模块是有独立命名空间的对象，包是架构边界。

## 今天必须记住的 Top10

**1. 模块**
一句话解释：作为模块对象加载的 .py 文件，有独立命名空间。
为什么存在：组织代码、隔离命名。
什么时候使用：拆分文件。
真实开发例子：`user_service.py` → module 对象。
一句代码：`import user_service`
容易踩坑：以为导入是把源码复制进来。
记忆口诀：模块是对象，非复制粘贴。

**2. 导入即执行**
一句话解释：导入会执行模块顶层代码。
为什么存在：需要构建模块命名空间。
什么时候使用：理解 import 副作用。
真实开发例子：顶层建连接会在导入时执行。
一句代码：`import config  # 跑顶层`
容易踩坑：顶层放重活。
记忆口诀：一导入，顶层就跑。

**3. sys.modules 缓存**
一句话解释：模块名→已加载对象的缓存，只执行一次。
为什么存在：避免重复执行、保持身份。
什么时候使用：理解「只执行一次」。
真实开发例子：`sys.modules["json"] is json`。
一句代码：`import sys; sys.modules[name]`
容易踩坑：模块级可变态被所有导入者共享。
记忆口诀：导过就缓存，复用同一份。

**4. 包**
一句话解释：按职责分组模块与子包。
为什么存在：清晰的导入边界。
什么时候使用：组织后端项目。
真实开发例子：`app/api /services /repositories`。
一句代码：`app/services/user_service.py`
容易踩坑：把包当纯文件夹。
记忆口诀：包是架构边界，非文件夹。

**5. `__init__.py`**
一句话解释：标记常规包，可定义包级导出。
为什么存在：包边界与公共 API。
什么时候使用：需要包级 re-export。
真实开发例子：轻量 re-export 公共名。
一句代码：`# app/__init__.py`
容易踩坑：`__init__` 里放连接/浏览器启动等重活。
记忆口诀：init 要轻，别放重活。

**6. 命名空间包**
一句话解释：3.3+ 可无 `__init__.py` 跨目录拆包。
为什么存在：跨发行分割逻辑包。
什么时候使用：真有打包需求时。
真实开发例子：插件式跨目录包。
一句代码：省略 `__init__.py`。
容易踩坑：普通应用滥用它。
记忆口诀：常规应用还是显式 init。

**7. `import module` vs `from module import name`**
一句话解释：前者保留命名空间，后者直接绑定对象。
为什么存在：可读性与依赖清晰度权衡。
什么时候使用：看哪个让所有权更明确。
真实开发例子：`from app.services.user import create_user`。
一句代码：`import json` / `from json import loads`
容易踩坑：过度用 from 隐藏来源。
记忆口诀：显式导入，来源清晰。

**8. `from module import *` 危险**
一句话解释：批量导入污染命名空间。
为什么存在：少数包导出场景。
什么时候使用：几乎不用。
真实开发例子：`import *` 引发名字碰撞。
一句代码：避免 `from x import *`。
容易踩坑：名字碰撞/遮蔽/隐藏依赖。
记忆口诀：星号导入=命名空间污染。

**9. 绝对 vs 相对导入**
一句话解释：大型后端优先绝对导入。
为什么存在：清晰、可移动。
什么时候使用：应用代码默认绝对。
真实开发例子：`from app.services.x import y`。
一句代码：避免 `from ...services import x`。
容易踩坑：深层相对导入难维护。
记忆口诀：大项目用绝对。

**10. 导入副作用**
一句话解释：导入时不应做连接/启动/调用等重活。
为什么存在：导入要「无聊」、可预测。
什么时候使用：设计模块顶层。
真实开发例子：定义工厂，运行期再建客户端。
一句代码：`def create_client(): return LLMClient()`
容易踩坑：顶层连数据库/启浏览器/调 LLM。
记忆口诀：导入定义工厂，运行才干活。

## 今天最重要的代码

```python
# sys.modules：导入只执行一次、身份稳定
import sys, json
print(sys.modules["json"] is json)   # True

# 顶层放工厂，避免导入副作用
# llm_client.py
class LLMClient: ...
def create_client() -> LLMClient:     # 运行期才建，导入时不连接
    return LLMClient()

# 显式绝对导入，避免 import *
from app.services.user_service import create_user   # 好
# from app.services.user_service import *           # 坏：污染命名空间
```

## 今天开发口诀

1. 模块是运行时对象，导入不是复制源码。
2. 导入会执行模块顶层代码。
3. `sys.modules` 缓存模块，通常只执行一次。
4. 别把请求/任务/会话状态放模块全局。
5. 包是架构边界，按职责划分。
6. `__init__.py` 保持轻量，不放连接/启动等重活。
7. 优先显式导入，避免 `from x import *`。
8. 大型后端优先绝对导入。
9. 导入应「无聊」：顶层只定义，不执行重活。
10. 循环导入是架构边界的警告信号。

## 今天最容易忘记的知识

1. **导入即执行**：为何忘——以为纯声明；正确 知顶层会跑；错误 顶层放重活；避免——用工厂。
2. **模块级可变态共享**：为何忘——像局部；正确 无状态；错误 模块 dict 存请求态；避免——依赖注入。
3. **import \* 污染**：为何忘——省事；正确 显式导入；错误 星号导入；避免——linter 禁用。
4. **只执行一次**：为何忘——以为每次跑；正确 记缓存；错误 期望重新执行；避免——记 sys.modules。
5. **深层相对导入**：为何忘——一时方便；正确 绝对导入；错误 `from ...x`；避免——大项目用绝对。
6. **init 放重活**：为何忘——想集中；正确 保持轻量；错误 init 里连库；避免——工厂延迟。
7. **循环导入**：为何忘——耦合渐增；正确 视作边界警告；错误 硬拆；避免——重构依赖方向。
8. **导入副作用破坏测试**：为何忘——本地能跑；正确 无副作用；错误 导入即连库；避免——运行期建。
9. **命名空间包滥用**：为何忘——省 init；正确 常规用显式；错误 到处省；避免——有理由才用。
10. **module vs package**：为何忘——概念糊；正确 文件=模块、目录=包；错误 混用；避免——记定义。

## 今天在 AI Backend 中的位置

```text
模块与包（架构边界 + 无副作用导入）
        ↓
FastAPI：app/api /services /repositories /schemas 分层包
        ↓
Playwright：browser 模块封装，导入不启动浏览器
        ↓
AI Backend：llm_client 顶层定义工厂，运行期才建客户端
        ↓
生产环境：可测试、启动可控、依赖方向清晰
```
上承作用域（全局态），下启 Day14 分层架构（包即层）。

## 一句话结束

导入即执行并缓存——让导入保持无聊，把重活留给运行期的工厂，把包当作架构边界。

# Day10 — Type Hints

## 一句话总结

类型标注是接口契约，默认不做运行时检查；FastAPI/Pydantic 会用它做校验。

## 今天必须记住的 Top10

**1. 类型标注 = 接口契约**
一句话解释：向人、IDE、静态检查、框架、AI 助手描述期望类型。
为什么存在：让边界契约显式。
什么时候使用：公共函数边界。
真实开发例子：`def create_user(email: str) -> User`。
一句代码：`def f(x: int) -> str: ...`
容易踩坑：以为它会强制运行时检查。
记忆口诀：标注是契约，非运行时锁。

**2. 默认不做运行时检查**
一句话解释：Python 本身不校验类型。
为什么存在：保持动态性能。
什么时候使用：明白校验靠谁。
真实开发例子：FastAPI/Pydantic 才做运行时校验。
一句代码：传错类型不会自动报错。
容易踩坑：以为标注=保证。
记忆口诀：标注给人看，校验靠框架。

**3. 参数类型标注**
一句话解释：标注公共参数类型。
为什么存在：调用者知道传什么。
什么时候使用：公共 API。
真实开发例子：`age: int`。
一句代码：`def f(email: str, age: int):`
容易踩坑：给显而易见的局部变量加噪声标注。
记忆口诀：公共参数要标，局部别噪。

**4. 返回类型标注**
一句话解释：标注返回类型，暴露缺失情况。
为什么存在：调用者知道拿到什么。
什么时候使用：所有公共函数。
真实开发例子：`-> User | None`。
一句代码：`def find(email: str) -> User | None:`
容易踩坑：声明 `-> User` 却返回 None。
记忆口诀：返回也要标，None 要诚实。

**5. 集合类型 `list[T]` / `dict[K,V]`**
一句话解释：标出元素类型，比裸 list 更有信息。
为什么存在：暴露元素形状。
什么时候使用：容器参数/返回。
真实开发例子：`list[User]`。
一句代码：`def send(users: list[User]) -> None:`
容易踩坑：用裸 `list`/`dict` 隐藏形状。
记忆口诀：容器带元素类型。

**6. `User | None` / Optional / Union**
一句话解释：`Optional[User] == User | None`。
为什么存在：显式表达「可能没有」。
什么时候使用：结果可能缺失。
真实开发例子：查无用户返 None。
一句代码：`-> User | None`
容易踩坑：巨大 Union 逼调用者处理太多分支。
记忆口诀：可缺就 `| None`，别堆大 Union。

**7. 类型推断**
一句话解释：工具能从上下文推断类型。
为什么存在：减少冗余标注。
什么时候使用：明显局部变量。
真实开发例子：`name = "Alice"` 推断 str。
一句代码：`count = 3  # int`
容易踩坑：空集合无法推断。
记忆口诀：能推断就少写，空容器要标。

**8. 空集合要标注**
一句话解释：空 list/dict 需显式标注。
为什么存在：无元素可推断。
什么时候使用：初始化空容器。
真实开发例子：`messages: list[ChatMessage] = []`。
一句代码：`data: dict[str, str] = {}`
容易踩坑：`items = []` 类型未知。
记忆口诀：空容器，手动标。

**9. `TypeVar`**
一句话解释：保留输入输出类型关系。
为什么存在：`object->object` 会丢类型。
什么时候使用：输入输出类型相关。
真实开发例子：`def identity(v: T) -> T`。
一句代码：`T = TypeVar("T")`
容易踩坑：用 object 丢失类型。
记忆口诀：`T->T` 保关系。

**10. `Generic`**
一句话解释：做可复用的泛型包装。
为什么存在：通用响应/容器。
什么时候使用：`Response[User]` 等。
真实开发例子：`Response[AgentResult]`。
一句代码：`class Response(Generic[T]): data: T`
容易踩坑：过度泛型化简单场景。
记忆口诀：泛型包装，一处定义处处复用。

## 今天最重要的代码

```python
from dataclasses import dataclass
from typing import Generic, TypeVar

T = TypeVar("T")

def find_user(email: str) -> "User | None":     # 诚实的返回契约
    ...

def send_emails(users: list["User"]) -> None:   # 容器带元素类型
    ...

def identity(value: T) -> T:                     # 保留输入输出关系
    return value

@dataclass
class Response(Generic[T]):                       # 可复用泛型包装
    data: T
    error: str | None = None

messages: list["ChatMessage"] = []                # 空集合要标注
```

## 今天开发口诀

1. 类型标注是接口契约，默认不做运行时检查。
2. FastAPI/Pydantic 才把标注用于运行时校验。
3. 标注公共参数与返回值，别给局部变量加噪声。
4. `-> User` 却可能返回 None 是不诚实契约，写 `User | None`。
5. 用 `list[User]` 而非裸 `list`，暴露元素形状。
6. `Optional[User]` 等价 `User | None`。
7. 避免巨大 Union 逼调用者处理太多分支。
8. 空集合需要显式标注类型。
9. 输入输出类型相关时用 `TypeVar` 保留关系。
10. 通用容器/响应用 `Generic`，别过度泛型化。

## 今天最容易忘记的知识

1. **不运行时检查**：为何忘——像强类型；正确 知靠框架；错误 以为会自动报错；避免——记默认不检查。
2. **`-> User` 返 None**：为何忘——多数返对象；正确 `User | None`；错误 隐藏 None；避免——诚实契约。
3. **裸 list/dict**：为何忘——省字；正确 `list[User]`；错误 隐藏元素；避免——带元素类型。
4. **空集合推断**：为何忘——以为能推；正确 手动标；错误 `items=[]`；避免——空容器标注。
5. **`object->object`**：为何忘——图省事；正确 `T->T`；错误 丢类型；避免——用 TypeVar。
6. **巨大 Union**：为何忘——都塞进去；正确 拆函数/建模型；错误 十路 Union；避免——收敛类型。
7. **过度局部标注**：为何忘——想严谨；正确 只标边界；错误 每个变量都标；避免——留白推断。
8. **Optional 语义**：为何忘——以为「可选参数」；正确 是「可能 None」；错误 混淆默认值；避免——记 `= None`。
9. **dict[str, object] 藏模型**：为何忘——图快；正确 建 model；错误 万能 dict；避免——领域建模。
10. **Generic 过度**：为何忘——炫技；正确 简单直接；错误 处处泛型；避免——按需。

类型标注是给人和框架看的接口契约——诚实标注边界，让 `User | None` 说真话。

# Day11 — Object-Oriented Programming

## 一句话总结

OOP 是职责设计：Is-A 用继承，Has-A 用组合；现代后端偏爱组合。

## 今天必须记住的 Top10

**1. 一切皆对象 / OOP=职责设计**
一句话解释：OOP 不是多写类，是把状态与行为归于清晰所有权。
为什么存在：管理复杂度。
什么时候使用：设计服务/领域对象。
真实开发例子：ChatService 聚合依赖。
一句代码：`class UserService: ...`
容易踩坑：为写类而写类。
记忆口诀：OOP 是职责设计，非类堆砌。

**2. 对象 / 类 / 实例**
一句话解释：类是蓝图，实例是产物，类本身也是对象。
为什么存在：批量创建同型对象。
什么时候使用：建模领域。
真实开发例子：`User` 类 → `user` 实例。
一句代码：`user = User()`
容易踩坑：混淆类与实例。
记忆口诀：类是模，实例是件。

**3. 状态 vs 行为**
一句话解释：状态是数据，行为是方法。
为什么存在：把相关数据与操作聚合。
什么时候使用：行为依赖状态时放进对象。
真实开发例子：服务持有依赖并暴露方法。
一句代码：`self.repo` + `def create(self)`
容易踩坑：无状态逻辑硬塞成类。
记忆口诀：有状态归对象，无状态用函数。

**4. `self`**
一句话解释：当前实例，约定非关键字。
为什么存在：方法访问实例状态。
什么时候使用：所有实例方法。
真实开发例子：`u1.say_hi()` → `User.say_hi(u1)`。
一句代码：`def say_hi(self): ...`
容易踩坑：漏写 self 触发 TypeError。
记忆口诀：self 是当前实例，自动传入。

**5. 类属性 vs 实例属性**
一句话解释：类属性共享，实例属性隔离。
为什么存在：区分共享值与每对象状态。
什么时候使用：真正共享才用类属性。
真实开发例子：`u1.company="G"` 只遮蔽 u1。
一句代码：`class U: company = "OpenAI"`
容易踩坑：可变类属性跨实例共享状态。
记忆口诀：类属性共享，实例属性各自。

**6. 属性/方法查找**
一句话解释：实例→类→父类→object。
为什么存在：确定性查找。
什么时候使用：理解 override。
真实开发例子：子类方法先命中。
一句代码：`Instance→Class→Parent→object`
容易踩坑：不懂查找顺序误判行为。
记忆口诀：从实例往上找，命中即停。

**7. 继承与 override**
一句话解释：继承表达 Is-A，子类同名方法覆盖父类。
为什么存在：复用与特化。
什么时候使用：真正 Is-A 关系。
真实开发例子：`Chromium(Browser)`。
一句代码：`class Chromium(Browser): def launch(self): ...`
容易踩坑：为复用而继承造成隐藏耦合。
记忆口诀：Is-A 才继承。

**8. `super()`**
一句话解释：委托到 MRO 中下一个类。
为什么存在：初始化父类状态。
什么时候使用：需要父类初始化时。
真实开发例子：`super().__init__(timeout)`。
一句代码：`super().__init__(...)`
容易踩坑：忘调 `super().__init__()` 父类状态未初始化。
记忆口诀：需父类初始化就 super。

**9. 组合 Has-A**
一句话解释：对象持有其它对象作为依赖。
为什么存在：低耦合、可替换、可测试。
什么时候使用：服务依赖。
真实开发例子：`ChatService(llm_client, cache)`。
一句代码：`self.llm_client = llm_client`
容易踩坑：用多继承拼装依赖。
记忆口诀：Has-A 用组合。

**10. MRO / 组合优于继承**
一句话解释：MRO 给确定性查找顺序；后端优先组合。
为什么存在：可维护、可替换。
什么时候使用：设计依赖结构。
真实开发例子：依赖注入替换 provider。
一句代码：构造函数注入依赖。
容易踩坑：滥用复杂多继承。
记忆口诀：组合优于继承。

## 今天最重要的代码

```python
class BaseClient:
    def __init__(self, timeout: int) -> None:
        self.timeout = timeout

class LLMClient(BaseClient):
    def __init__(self, timeout: int, model: str) -> None:
        super().__init__(timeout)     # 必须显式调用，否则父类状态未初始化
        self.model = model

# 组合优于继承（Has-A）
class ChatService:
    def __init__(self, llm_client: LLMClient, cache: "RedisCache") -> None:
        self.llm_client = llm_client   # 依赖注入：可替换、可测试
        self.cache = cache
```

## 今天开发口诀

1. OOP 是职责设计，不是多写类。
2. 类是蓝图、实例是产物，类本身也是对象。
3. 行为依赖状态时放进对象，无状态逻辑用函数。
4. `self` 是自动传入的当前实例，实例方法别漏。
5. 类属性共享、实例属性隔离，慎用可变类属性。
6. 属性查找：实例→类→父类→object，命中即停。
7. Is-A 用继承，Has-A 用组合。
8. 需要父类初始化时别忘 `super().__init__()`。
9. 现代后端优先组合 + 依赖注入。
10. 别在应用代码里滥用复杂多继承。

## 今天最容易忘记的知识

1. **漏 self**：为何忘——写惯脚本；正确 实例方法带 self；错误 忘写；避免——记方法签名。
2. **可变类属性**：为何忘——写着方便；正确 实例属性存状态；错误 类级 list 共享；避免——每对象独立。
3. **漏 super().__init__**：为何忘——以为自动；正确 显式调用；错误 父类状态未初始化；避免——记不自动。
4. **为复用而继承**：为何忘——省代码；正确 组合；错误 继承拿方法；避免——Is-A 判断。
5. **Is-A vs Has-A**：为何忘——都能建模；正确 关系判断；错误 混用；避免——记两词。
6. **override 查找**：为何忘——不懂顺序；正确 子类先命中；错误 误判行为；避免——记查找链。
7. **类 vs 实例属性遮蔽**：为何忘——赋值像改类；正确 只遮蔽该实例；错误 以为改了类；避免——记 shadowing。
8. **多继承 MRO**：为何忘——少用；正确 简单继承；错误 复杂菱形；避免——避免滥用。
9. **服务有状态**：为何忘——图方便；正确 无状态服务；错误 存请求态；避免——依赖注入（Day14）。
10. **self 非关键字**：为何忘——像关键字；正确 是约定；错误 以为必须叫 self；避免——理解本质。

## 今天在 AI Backend 中的位置

```text
OOP（职责设计 + 组合 + 依赖注入）
        ↓
FastAPI：FastAPI()/Request/Response 对象、Service 层组合
        ↓
Playwright：Browser→Context→Page→Locator，共享行为、隔离状态
        ↓
AI Backend：ChatService 组合 LLMClient/PromptBuilder/VectorStore/Cache
        ↓
生产环境：provider 可替换、服务可测试、Agent 行为模块化
```
上承类型标注（对象契约），下启 Day14 架构（服务对象组装）。

## 一句话结束

OOP 是把状态与行为归于清晰所有权——Is-A 才继承，Has-A 用组合，依赖显式注入。

# Day12 — Context Managers

## 一句话总结

上下文管理器保证资源确定性释放，即使函数体抛异常；业务不该拥有资源管理。

## 今天必须记住的 Top10

**1. 资源生命周期 Acquire/Use/Release**
一句话解释：获取→使用→释放，释放必须保证。
为什么存在：Release 是工程师最常忘的一步。
什么时候使用：文件/连接/上下文/流/锁。
真实开发例子：DB session 用后关闭。
一句代码：`with open(p) as f: ...`
容易踩坑：漏了失败路径的释放。
记忆口诀：拿→用→还，还是关键。

**2. 业务不该拥有资源管理**
一句话解释：上下文管理器负责获取释放，业务只负责使用。
为什么存在：职责分离。
什么时候使用：任何资源操作。
真实开发例子：handler 只写业务，session 由依赖管理。
一句代码：业务只用 db，不管开关。
容易踩坑：业务逻辑里手动 close。
记忆口诀：业务只用，管理归 with。

**3. `try / finally`**
一句话解释：finally 成功失败都执行，保证释放。
为什么存在：确定性清理。
什么时候使用：手动资源管理。
真实开发例子：`finally: conn.close()`。
一句代码：`try: ... finally: close()`
容易踩坑：把 close 放在业务之后而非 finally。
记忆口诀：finally 必执行。

**4. `with` 语句**
一句话解释：`try/finally` 的简洁语法且保证清理。
为什么存在：更安全更清晰。
什么时候使用：所有资源获取。
真实开发例子：`with open(...) as f`。
一句代码：`with res() as r: ...`
容易踩坑：不用 with 手动开关漏关。
记忆口诀：with 一包，清理自动。

**5. `__enter__` / `__exit__`**
一句话解释：enter 获取并返回 as 目标，exit 释放并收异常信息。
为什么存在：上下文管理协议。
什么时候使用：自定义资源类。
真实开发例子：FileManager。
一句代码：`def __exit__(self, t, v, tb): ...`
容易踩坑：`__exit__` 误返回 True 吞异常。
记忆口诀：enter 拿、exit 还。

**6. `__exit__` 返回值语义**
一句话解释：返回 False 异常传播（正常），True 抑制异常（罕见）。
为什么存在：控制异常是否被吞。
什么时候使用：极少数需要抑制时。
真实开发例子：默认 return False。
一句代码：`return False`
容易踩坑：误返回 True 静默吞错。
记忆口诀：清理照做，异常照抛。

**7. `@contextmanager` + yield**
一句话解释：用生成器写上下文管理器，yield 前获取、后释放。
为什么存在：更轻量的写法。
什么时候使用：简单资源封装。
真实开发例子：`open_file` 生成器。
一句代码：`@contextmanager def f(): ... yield r ... finally: ...`
容易踩坑：yield 不在 try/finally 里，失败时泄漏。
记忆口诀：yield 必裹 try/finally。

**8. yield vs return（清理相位）**
一句话解释：return 无恢复相位，yield 可在恢复后清理。
为什么存在：清理需要 resume 阶段。
什么时候使用：需要后置清理。
真实开发例子：yield 后 finally 关闭。
一句代码：`yield r` 之后 finally 清理。
容易踩坑：用 return 导致无处清理。
记忆口诀：yield 能续，return 落幕。

**9. FastAPI yield 依赖**
一句话解释：建 session→yield→finally 关闭，请求作用域。
为什么存在：请求结束保证清理。
什么时候使用：DB session 依赖。
真实开发例子：`get_db` 依赖。
一句代码：`try: yield db finally: db.close()`
容易踩坑：清理没放 finally，异常时泄漏。
记忆口诀：yield 依赖，finally 收尾。

**10. 多资源反序释放**
一句话解释：多个资源按获取的反序释放。
为什么存在：正确的依赖清理顺序。
什么时候使用：组合多个上下文。
真实开发例子：`with get_db() as db, get_redis() as cache:`。
一句代码：嵌套/并列 with。
容易踩坑：忘关 LLM 流导致挂起 socket。
记忆口诀：后拿的先还。

## 今天最重要的代码

```python
from contextlib import contextmanager

@contextmanager
def open_file(path: str):
    f = open(path)          # Acquire
    try:
        yield f             # Use（必须在 try 内）
    finally:
        f.close()           # Release（成功失败都执行）

# FastAPI yield 依赖：请求作用域，异常也清理
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()

# Playwright：共享 Browser，隔离并关闭 Context
context = await browser.new_context()
try:
    page = await context.new_page()
    ...
finally:
    await context.close()
```

## 今天开发口诀

1. 上下文管理器保证资源确定性释放，即使抛异常。
2. 业务只负责 Use，Acquire/Release 归上下文管理器。
3. `finally` 成功失败都执行，把释放放这里。
4. `with` 是更安全的 `try/finally`。
5. `__enter__` 获取并返回 as 目标，`__exit__` 释放。
6. `__exit__` 返回 True 会吞异常，通常别这么做。
7. `@contextmanager` 里 `yield` 必须包在 `try/finally`。
8. 清理需要恢复相位，所以用 yield 不用 return。
9. FastAPI yield 依赖在 finally 关闭 session。
10. Playwright 共享 Browser、每 job 隔离并关闭 Context。

## 今天最容易忘记的知识

1. **close 放业务后**：为何忘——顺手写；正确 放 finally；错误 失败时漏关；避免——用 with。
2. **yield 未裹 try/finally**：为何忘——看着对；正确 包住；错误 异常泄漏；避免——固定模板。
3. **`__exit__` 返回 True**：为何忘——误写；正确 return False；错误 静默吞异常；避免——默认 False。
4. **共享 session/context**：为何忘——省资源；正确 每 job 隔离；错误 跨 job 共享；避免——作用域隔离。
5. **忘关 LLM 流**：为何忘——隐式；正确 上下文管理；错误 挂起 socket 浪费 token；避免——with 包裹。
6. **业务管清理**：为何忘——就近写；正确 上下文管理器管；错误 逻辑变清理坏；避免——职责分离。
7. **多资源顺序**：为何忘——不留意；正确 反序释放；错误 顺序乱；避免——嵌套 with。
8. **锁不释放**：为何忘——只想加锁；正确 finally 释放；错误 异常时死锁；避免——with 锁。
9. **enter 返回值**：为何忘——以为返 self；正确 返 as 目标；错误 返错对象；避免——记协议。
10. **startup/shutdown**：为何忘——只想请求级；正确 lifespan 管全局资源；错误 泄漏；避免——用 lifespan。

## 今天在 AI Backend 中的位置

```text
上下文管理器（确定性资源清理）
        ↓
FastAPI：yield 依赖管理 DB session、lifespan 管启动/关闭
        ↓
Playwright：每 job 关闭 BrowserContext
        ↓
AI Backend：LLM 流、Redis 连接、DB session、并发锁都用 with 包裹
        ↓
生产环境：无资源泄漏、无挂起 socket、无连接池耗尽
```
上承生成器（yield 机制），下启 Day13 异步（异步资源清理）。

## 一句话结束

清理永远要执行、异常通常要上抛——把 Acquire/Release 交给上下文管理器，业务只管 Use。

# Day13 — Async Programming

## 一句话总结

单线程事件循环重叠 I/O 等待——异步提升 I/O 吞吐，不提升 CPU 算力。

## 今天必须记住的 Top10

**1. I/O 密集 vs CPU 密集**
一句话解释：异步只适合 I/O 密集。
为什么存在：异步靠重叠等待。
什么时候使用：网络/DB/文件等待。
真实开发例子：并发调多个 API。
一句代码：I/O 用 async，CPU 用进程/线程。
容易踩坑：用异步跑 CPU 密集。
记忆口诀：异步治等待，不治计算。

**2. 阻塞 vs 非阻塞**
一句话解释：`time.sleep` 冻结整个事件循环，`asyncio.sleep` 只挂起本任务。
为什么存在：单线程一次一行。
什么时候使用：区分阻塞调用。
真实开发例子：async 里误用阻塞驱动拖垮全部请求。
一句代码：`await asyncio.sleep(2)`
容易踩坑：`async def` 里调阻塞函数。
记忆口诀：async 里禁 time.sleep。

**3. 事件循环**
一句话解释：单线程协作式调度器，在每个 await 切任务。
为什么存在：用重叠等待实现并发。
什么时候使用：理解并发本质。
真实开发例子：一任务 await 就让出循环。
一句代码：`Ready Queue→loop→run→await→suspend`
容易踩坑：以为是并行 CPU。
记忆口诀：并发=重叠等待，非并行算力。

**4. 协程 vs 协程对象**
一句话解释：调用协程函数只生成计划，不执行体。
为什么存在：把执行交给循环。
什么时候使用：区分定义与调度。
真实开发例子：`coro = hello()` 未执行。
一句代码：`coro = hello()  # 计划`
容易踩坑：调协程却不 await（体不运行）。
记忆口诀：调用只造计划，await 才跑。

**5. Task vs 协程**
一句话解释：Task 是被循环并发驱动的协程。
为什么存在：真正并发要 create_task。
什么时候使用：需要并发。
真实开发例子：`asyncio.create_task(coro)`。
一句代码：`task = asyncio.create_task(coro)`
容易踩坑：只 `await coro()` 变成顺序执行。
记忆口诀：要并发就 create_task。

**6. `await`**
一句话解释：挂起当前协程、让出循环、就绪后恢复。
为什么存在：非阻塞等待。
什么时候使用：等待协程/Task/future。
真实开发例子：`await call_api()`。
一句代码：`result = await coro`
容易踩坑：以为 await 开线程。
记忆口诀：await 让出循环，不开线程。

**7. `asyncio.gather()`**
一句话解释：并发运行多个协程，按输入顺序返回结果。
为什么存在：批量并发。
什么时候使用：并发多个 I/O。
真实开发例子：总时约 max 而非 sum。
一句代码：`a, b = await asyncio.gather(t1(), t2())`
容易踩坑：无上限 gather 外部调用引发 429。
记忆口诀：gather 并发，结果按输入序。

**8. 任务取消**
一句话解释：`cancel()` 是请求，在下个 await 注入 CancelledError。
为什么存在：协作式取消。
什么时候使用：客户端断开/超时。
真实开发例子：FastAPI 客户端断开取消请求任务。
一句代码：`try: await work() except CancelledError: ...; raise`
容易踩坑：以为 cancel 立即杀死。
记忆口诀：取消是请求，非即杀。

**9. 异常传播**
一句话解释：Task 存住异常，直到被 await 才抛。
为什么存在：延迟到检索点。
什么时候使用：并发任务错误处理。
真实开发例子：不 await 失败任务 → 异常丢失。
一句代码：`await task  # 此处重抛`
容易踩坑：fire-and-forget 任务失败无声。
记忆口诀：不 await，异常丢。

**10. Semaphore**
一句话解释：限制并发数以保护下游容量。
为什么存在：防 429/超时/连接池耗尽。
什么时候使用：调外部 API/DB/GPU。
真实开发例子：`Semaphore(10)` 限 OpenAI 并发。
一句代码：`async with sem: await call()`
容易踩坑：追求最大并发而非稳定吞吐。
记忆口诀：限并发，稳吞吐。

## 今天最重要的代码

```python
import asyncio

# 阻塞 vs 非阻塞
# time.sleep(2)              # 冻结整个事件循环 —— 禁止
await asyncio.sleep(2)       # 只挂起本任务

# 协程 → Task → 并发
async def hello(): ...
coro = hello()                        # 计划，未执行
task = asyncio.create_task(coro)      # 调度并发
await task

# gather 并发（总时≈max），按输入顺序返回
a, b = await asyncio.gather(call1(), call2())

# Semaphore 限并发保护下游
sem = asyncio.Semaphore(10)
async def call(p):
    async with sem:
        return await call_openai(p)

# 不可避免的阻塞工作丢给线程
result = await asyncio.to_thread(blocking_fn, arg)
```

## 今天开发口诀

1. 异步提升 I/O 吞吐，不提升 CPU 算力。
2. `async def` 里绝不调 `time.sleep` 或阻塞驱动。
3. 事件循环单线程协作式，在每个 await 切任务。
4. 调用协程函数只生成协程对象，不执行体。
5. 真正并发用 `create_task`，只 await 是顺序执行。
6. `await` 让出循环，不创建线程。
7. `gather` 并发运行、按输入顺序返回结果。
8. 外部调用的 `gather` 要用 `Semaphore` 限并发。
9. await 你的 Task，异常才会浮现。
10. 取消是协作式的，`CancelledError` 在下个 await 抛出。

## 今天最容易忘记的知识

1. **time.sleep 冻结循环**：为何忘——习惯性；正确 `asyncio.sleep`；错误 async 里 time.sleep；避免——记单线程。
2. **调协程不 await**：为何忘——像调用；正确 await/create_task；错误 体不执行 + 警告；避免——记计划 vs 执行。
3. **只 await 无并发**：为何忘——以为并发；正确 create_task；错误 顺序执行；避免——记区别。
4. **无上限 gather**：为何忘——图快；正确 Semaphore；错误 429/池耗尽；避免——限并发。
5. **fire-and-forget 丢异常**：为何忘——不 await；正确 await 或 done callback；错误 静默丢失；避免——检索异常。
6. **cancel 非即杀**：为何忘——直觉；正确 协作式；错误 期望立即停；避免——记下个 await。
7. **阻塞驱动**：为何忘——同步库顺手；正确 async 库/to_thread；错误 阻塞全部请求；避免——用异步 API。
8. **CancelledError 清理**：为何忘——只想停；正确 except/finally 清理再 raise；错误 吞掉；避免——重抛。
9. **CPU 密集用异步**：为何忘——以为万能；正确 进程/线程；错误 异步跑计算；避免——分清类型。
10. **await 开线程误解**：为何忘——直觉；正确 只让出循环；错误 以为并行；避免——记单线程。

## 今天在 AI Backend 中的位置

```text
异步（事件循环重叠 I/O 等待）
        ↓
FastAPI：async 端点、to_thread 处理阻塞、客户端断开取消任务
        ↓
Playwright：Semaphore 限制并发浏览器
        ↓
AI Backend：并发 LLM 调用 + Semaphore 限流 + 重试/退避
        ↓
生产环境：稳定吞吐、保护下游、无池耗尽
```
上承上下文管理器（异步清理），下启 Day14 架构的并发与扩展。

## 一句话结束

异步是单线程重叠等待——优化的是稳定吞吐而非最大并发，`async` 里永远别阻塞事件循环。

# Day14 — Mini Project & Backend Architecture

## 一句话总结

跑通一次是脚本，分层清晰、职责单一才是系统；把 Day01–13 整合为分层 AI 后端。

## 今天必须记住的 Top10

**1. 分层架构**
一句话解释：API→Service→Browser→LLM→Repository→DB，数据向下流。
为什么存在：职责分离、可测试、可维护。
什么时候使用：设计后端项目。
真实开发例子：抓取→总结→保存流水线。
一句代码：`Router→Service→Repository`
容易踩坑：跨层泄漏（Router 写 SQL）。
记忆口诀：数据向下流，依赖指接口。

**2. API/Router 层（薄）**
一句话解释：校验请求 + 委托 service，不含业务逻辑。
为什么存在：传输层保持薄。
什么时候使用：定义端点。
真实开发例子：`@router.post` 接模型返模型。
一句代码：`return await svc.summarize(req.url)`
容易踩坑：路由里写业务。
记忆口诀：路由只校验和转发。

**3. Service 层**
一句话解释：编排工作流，不懂 HTTP/SQL。
为什么存在：业务核心、无状态。
什么时候使用：组织业务流程。
真实开发例子：extract→summarize→save。
一句代码：`class SummaryService:`
容易踩坑：service 里塞 HTTP/SQL。
记忆口诀：service 编排，不碰传输持久。

**4. Browser 层**
一句话解释：封装 Playwright，返回纯数据，finally 关 context。
为什么存在：基础设施隔离。
什么时候使用：抓取网页。
真实开发例子：返回文本而非 FastAPI 模型。
一句代码：`return page_text`
容易踩坑：返回 FastAPI 模型/JSON。
记忆口诀：浏览器返数据，不返模型。

**5. LLM 层**
一句话解释：接口隐藏 provider，支持多provider与假实现测试。
为什么存在：可故障转移、可测试。
什么时候使用：调用大模型。
真实开发例子：OpenAI/Anthropic/Gemini 互换。
一句代码：`class LLMClient(Protocol):`
容易踩坑：LLM 层懂 HTTP 细节。
记忆口诀：provider 藏接口后。

**6. Repository 层**
一句话解释：隐藏数据库，返回领域对象而非原始行。
为什么存在：service 不写 SQL。
什么时候使用：读写持久化。
真实开发例子：`save_task/get_task`。
一句代码：`async def get_task(self, id) -> Task | None`
容易踩坑：SQL 向上泄漏。
记忆口诀：仓储藏 DB，返领域对象。

**7. 依赖注入**
一句话解释：从外部传入依赖，不在内部构造。
为什么存在：可测试、可替换、显式。
什么时候使用：组装服务。
真实开发例子：`Depends(get_service)`。
一句代码：`def __init__(self, repo): self.repo = repo`
容易踩坑：服务内部 new 依赖。
记忆口诀：依赖传进来，别自己造。

**8. 无状态服务**
一句话解释：实例不存每请求状态，任何副本可服务任何请求。
为什么存在：并发安全、水平扩展。
什么时候使用：所有服务对象。
真实开发例子：多副本负载均衡。
一句代码：不在 self 上存请求态。
容易踩坑：共享可变状态 → 数据泄漏/竞态。
记忆口诀：无状态才可横向扩。

**9. Worker vs Async**
一句话解释：async 给单 worker I/O 吞吐，worker/副本给并行与隔离。
为什么存在：不同扩展维度。
什么时候使用：容量规划。
真实开发例子：长任务用队列+worker+任务状态。
一句代码：返回 task_id，worker 处理。
容易踩坑：30 秒阻塞请求做长任务。
记忆口诀：async 提吞吐，worker 提容量。

**10. Semaphore / 重试 / 指数退避**
一句话解释：限并发到下游容量，429 用指数退避重试。
为什么存在：保护下游、稳定吞吐。
什么时候使用：调外部限流服务。
真实开发例子：找瓶颈（OpenAI 限流/DB 池/浏览器内存）。
一句代码：退避 `wait = base * 2 ** attempt`。
容易踩坑：优化最大并发而非稳定吞吐。
记忆口诀：限并发、退避重试、稳吞吐。

## 今天最重要的代码

```python
# 薄路由：校验 + 委托
@router.post("/summarize", response_model=SummaryResponse)
async def summarize(req: SummarizeRequest, svc: SummaryService = Depends(get_service)):
    return await svc.summarize(req.url)     # 无业务逻辑

# 无状态 service，依赖注入，编排工作流
class SummaryService:
    def __init__(self, browser: "BrowserClient", llm: "LLMClient", repo: "TaskRepository") -> None:
        self.browser, self.llm, self.repo = browser, llm, repo
    async def summarize(self, url: str) -> "SummaryResponse":
        text = await self.browser.extract(url)      # Browser 层：返纯数据
        summary = await self.llm.summarize(text)    # LLM 层：藏 provider
        task = await self.repo.save_task(url, summary)  # Repository：藏 DB
        return SummaryResponse(task_id=task.id, summary=summary)

# Repository 隐藏数据库，返回领域对象
class TaskRepository:
    async def save_task(self, url: str, summary: str) -> "Task": ...
    async def get_task(self, task_id: int) -> "Task | None": ...
```

## 今天开发口诀

1. 跑通一次是脚本，分层清晰才是系统。
2. 数据向下流，依赖指向接口而非实现。
3. Router 薄：校验请求 + 委托，不写业务。
4. Service 编排工作流，不懂 HTTP 或 SQL。
5. Browser/LLM 是接口后的基础设施，返回数据不返模型。
6. Repository 隐藏数据库，返回领域对象。
7. 依赖注入让服务可测试、可替换。
8. 无状态服务才能水平扩展。
9. async 提单 worker 吞吐，worker/副本提容量。
10. 长任务返回 task_id 交 worker，别阻塞 30 秒请求。

## 今天最容易忘记的知识

1. **路由写业务**：为何忘——就近；正确 委托 service；错误 路由里编排；避免——薄路由。
2. **service 写 SQL**：为何忘——省层；正确 走 repository；错误 service 直连 DB；避免——分层。
3. **共享可变状态**：为何忘——图快；正确 无状态；错误 self 存请求态；避免——依赖注入。
4. **内部 new 依赖**：为何忘——顺手；正确 注入；错误 服务内构造；错误难测；避免——DI。
5. **browser 返模型**：为何忘——省转换；正确 返纯数据；错误 返 FastAPI 模型；避免——基础设施隔离。
6. **无限并发**：为何忘——图快；正确 Semaphore；错误 打爆下游；避免——限并发。
7. **阻塞长请求**：为何忘——简单；正确 队列+worker+状态；错误 30 秒阻塞；避免——异步任务。
8. **无重试退避**：为何忘——乐观；正确 指数退避；错误 遇 429 直接失败；避免——退避重试。
9. **provider 硬编码**：为何忘——一个够用；正确 接口抽象；错误 绑死 OpenAI；避免——多 provider。
10. **不找瓶颈**：为何忘——盲目并发；正确 定位瓶颈；错误 乱加并发；避免——吞吐分析。

```text
        ↓

        ↓
```
